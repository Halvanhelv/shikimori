# 86 — Plug-конвейер (Elixir): обработка запроса по шагам

## Простыми словами
В neko HTTP-запрос проходит через цепочку «плагов» — маленьких шагов: логирование → проверка токена → разбор JSON → поиск маршрута → выполнение. Каждый шаг получает запрос, что-то делает и передаёт дальше. Это как Rack-middleware в Rails, только явный и упорядоченный.

## Реальный код
```elixir
defmodule Neko.Router do
  use Plug.Router

  plug Plug.Logger                                  # 1. лог
  plug Neko.Plug.Authenticate, token: @token        # 2. проверка токена (см. 55)
  plug :match                                        # 3. найти маршрут
  plug Plug.Parsers, parsers: [:json]                # 4. разобрать тело
  plug :dispatch                                     # 5. выполнить обработчик

  post "/user_rate" do
    request = Neko.Request.new(conn.body_params)
    diff = request |> Neko.UserHandler.process()
    send_resp(conn, 201, Poison.encode!(diff))
  end
end
```

## Как работает
- Каждый `plug` — шаг, получает `conn` (соединение), возвращает изменённый `conn`.
- Шаги выполняются по порядку сверху вниз.
- Любой шаг может оборвать цепочку (`Authenticate` шлёт 401 и `halt`, см. [49](49-policy-static-facade.md) идею gate).
- `:dispatch` запускает обработчик найденного маршрута.

## 🆚 Классический Rails
**Аналогия:** это как Rack-middleware + `before_action` в Rails, но **явный список шагов** в одном месте, а не разбросанные фильтры по контроллерам.
**Польза:** видно весь конвейер обработки запроса сверху вниз; порядок явный; шаг переиспользуется в любом роутере.

## Бенефиты
- Прозрачный, упорядоченный конвейер обработки.
- Шаги маленькие, переиспользуемые, тестируемые.
- Ранний обрыв (auth) до тяжёлой работы.

## Когда полезно вне Elixir
Идея та же, что Rack-middleware и [контроллер-concern'ы](45-controller-concerns.md): выделяй сквозные шаги (auth, парсинг, лог) в переиспользуемые звенья конвейера, а не копируй в каждый обработчик.
