# 77 — `apipie` (документация API из кода)

## Простыми словами
Документация API пишется прямо над методами контроллера. Из этих описаний автоматически генерится страница с описанием эндпоинтов, параметров, примеров — и она не устаревает, потому что лежит рядом с кодом.

## Реальный код
```ruby
class Api::V1::UserRatesController < Api::V1Controller
  api :POST, '/user_rates/:id/increment', 'Increment episodes/chapters by 1'
  description 'Requires `user_rates` oauth scope'
  param :id, :number, required: true
  def increment
    # ...
  end
end
```

## Как работает
- Макросы `api`, `param`, `description` над экшеном описывают эндпоинт.
- apipie собирает их в живую HTML-документацию (`/api/doc`).
- `param` ещё и **валидирует** вход: неверный тип/отсутствие обязательного → ошибка.
- Тесты могут записывать примеры ответов (`APIPIE_RECORD=all`).

## 🆚 Классический Rails
**Обычно:** документацию API ведут отдельно (Notion/Swagger-файл вручную) → она отстаёт от кода, врёт. Валидацию параметров пишут руками в каждом экшене.
**Здесь:** доки и валидация — рядом с кодом, из одного источника. Изменил параметр → доки обновились.

## Бенефиты
- Документация не устаревает (живёт в коде).
- Описание = валидация входа (двойная польза).
- Примеры ответов записываются из тестов.

## Когда не надо
Внутренний API без внешних потребителей — доки могут быть избыточны. Альтернативы: OpenAPI/Swagger-генераторы, rswag (через тесты).
