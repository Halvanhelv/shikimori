# 68 — jsonb-колонка + value-объект

## Простыми словами
Сложное значение (например «неполная дата»: год есть, месяц неизвестен) хранят в колонке-jsonb, а в коде заворачивают в умный объект, который умеет это разбирать и сравнивать.

## Реальный код
```ruby
class IncompleteDate          # см. 06
  include ShallowAttributes
  include Comparable
  attribute :year, NilInteger, allow_nil: true
  attribute :month, NilInteger, allow_nil: true
  attribute :day, NilInteger, allow_nil: true
end

# модель хранит это в jsonb-колонке и кастует к объекту:
# aired_on: { "year": 2019, "month": null, "day": null } (в БД)
anime.aired_on.year      # 2019
anime.aired_on.human     # "2019" (месяц неизвестен — не показываем)
```

## Как работает
- В БД — колонка `jsonb` (произвольная JSON-структура).
- Кастомный тип/value-объект ([06](06-shallow-attributes.md)) сериализует объект в JSON при записи и собирает обратно при чтении.
- Объект несёт логику (сравнение, человекочитаемый вывод), а не голый хеш.

## 🆚 Классический Rails
**Обычно:** или три отдельные колонки (`year`, `month`, `day` — раздувают таблицу), или голый хеш в тексте (`row.data['year']` — без логики, без валидации, легко опечататься в ключе).
**Здесь:** одна jsonb-колонка + типизированный объект с методами. Структура гибкая, логика в одном месте, доступ через `.year`, а не `['year']`.

## Бенефиты
- Гибкая вложенная структура без новых колонок/миграций.
- Логика значения (сравнение, форматирование) инкапсулирована.
- jsonb индексируется и запрашивается в Postgres при необходимости.

## Когда не надо
Если по полю часто фильтруешь/джойнишь как по обычному столбцу — лучше отдельная колонка с индексом. jsonb хорош для «структура целиком, читается как объект».
