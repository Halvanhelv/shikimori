# 84 — ExConstructor (Elixir): структура из «грязного» хеша

## Простыми словами
Когда neko получает JSON от shikimori, ключи могут быть строками или атомами, в разном стиле. ExConstructor собирает из такого хеша аккуратную типизированную структуру, не разбираясь руками с каждым ключом.

## Реальный код
```elixir
defmodule Neko.Request do
  defstruct ~w(id user_id target_id score action status episodes)a

  use ExConstructor, atoms: true, strings: true,   # принимает и атомы, и строки-ключи
    camelcase: false, underscore: false

  # из тела HTTP-запроса:
  request = Neko.Request.new(conn.body_params)   # хеш → %Neko.Request{}
end
```

## Как работает
- `defstruct` объявляет поля структуры (аналог «модели без БД»).
- `use ExConstructor` генерит `.new(map)`, который кладёт значения из хеша в поля структуры, понимая разные форматы ключей (строка/атом/camelCase).
- Дальше работаешь с типизированной `%Neko.Request{}`, а не с сырым хешем.

## 🆚 «Сырой хеш» подход
**Обычно:** таскают сырой map из JSON (`params["user_id"]`) — легко опечататься в ключе, нет проверки полей, формат ключей надо помнить.
**Здесь:** один вызов `.new` → строгая структура с известными полями (`request.user_id`). Аналог Ruby — `ShallowAttributes`/`Data.define` ([06](06-shallow-attributes.md)).

## Бенефиты
- Граница «грязный JSON → чистая структура» в одном месте.
- Доступ через поля, а не строковые ключи (меньше опечаток).
- Терпим к формату ключей из внешнего источника.

## Когда полезно вне neko
Любая граница с внешними данными (API, очередь, форма): сразу заворачивай сырой ввод в типизированный объект, дальше работай только с ним.
