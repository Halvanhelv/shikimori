# 18 — Chewy: Elasticsearch с языковыми анализаторами

## Простыми словами
Обычный поиск по базе тупой: «narto» не найдёт «Naruto», а русское название — английское. Elasticsearch умный: понимает опечатки, транслит, разные языки и синонимы. Chewy — это «мостик» между моделями Rails и Elasticsearch: описываешь, какие поля и как индексировать, а синхронизация идёт автоматически. Как гугл, который находит, что ты имел в виду, даже с ошибками — только для аниме.

## Что
Поиск аниме по названиям на 4 языках + синонимы + транслит + опечатки. `app/chewy/application_index.rb` (~442 строки): свои анализаторы, японская романизация, ngram, кастомный скоринг.

## Реальный код
```ruby
class AnimesIndex < ApplicationIndex
  NAME_FIELDS = %i[name russian english japanese synonyms_0 ... license_name_ru]

  define_type Anime do
    NAME_FIELDS.each do |name_field|
      field name_field, type: 'keyword', index: false,
        value: ->(model) { ... },
        fields: {                          # ОДНО значение → 4 анализа (мульти-поля)
          original: ORIGINAL_FIELD,        # точное совпадение
          edge_phrase: EDGE_PHRASE_FIELD,  # автодополнение фразы
          edge_word: EDGE_WORD_FIELD,      # автодополнение слова
          ngram: NGRAM_FIELD               # опечатки/куски
        }
    end
  end
end

# синк модели → индекс только при изменении искомых полей:
update_index('animes#anime') do
  self if saved_change_to_name? || saved_change_to_russian? || saved_change_to_score?
end
```

## Как работает
- **Мульти-поля**: одно `name` индексируется 4 анализаторами сразу; на запрос ищем по всем и комбинируем скор (точное > ngram).
- `char_filter` романизирует японский на токенизации (chi→ti, sha→sya) → «shana» = «syana».
- Своя `scripted_tfidf` similarity: IDF отключён, в поиске по названиям он размывает скор.
- `update_index` срабатывает только если поменялось искомое поле.

## Зачем / где полезно
Мультиязычный полнотекстовый поиск, опечатки/транслит, автодополнение. Учит: индекс = настраиваемый pipeline токенизации + кастомный скоринг, а не «просто полнотекст».

## 🆚 Классический Rails
**Обычно:** поиск делают через `WHERE name ILIKE '%...%'` в Postgres — медленно на больших таблицах (нет индекса под `%...%`), не находит опечатки/транслит/синонимы, не ранжирует по релевантности.
**Здесь:** отдельный поисковый движок (Elasticsearch) с настроенными анализаторами и скорингом; модель синкается в индекс автоматически через `update_index` при изменении искомых полей.

## Бенефиты
- Находит с опечатками, на разных языках, по транслиту и синонимам.
- Релевантность настраивается (свой скоринг), а не алфавит/случайный порядок.
- Быстро на больших объёмах (инвертированный индекс), не грузит основную БД.
- Синхронизация модель→индекс автоматическая и точечная (только нужные поля).
