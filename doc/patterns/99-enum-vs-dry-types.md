# 99 — Rails enum vs dry-types enum: что выбрать

## Простыми словами
«Статус может быть только из списка» можно сделать двумя способами: встроенным Rails-enum (хранит число, даёт хелперы) или строгим dry-types enum (валидирует на входе, один источник значений). В shikimori чаще второй — разбираемся почему.

## Реальный код
```ruby
# Rails enum — маппинг строка↔число, хелперы:
class Order < ApplicationRecord
  enum status: { pending: 0, paid: 1, shipped: 2 }
end
order.paid!; order.pending?

# dry-types + enumerize — строгий набор, значения из одного места (см. 05, 31):
module Types::Anime
  Status = Types::Strict::Symbol.constructor(&:to_sym).enum(:anons, :ongoing, :released)
end
class Anime < ApplicationRecord
  enumerize :status, in: Types::Anime::Status.values, predicates: true
end
```

## Как работает / сравнение
| | Rails `enum` | dry-types + `enumerize` |
|--|--|--|
| Хранение | целое число (маппинг) | строка/символ как есть |
| Валидация мусора | молча принимает невалидное местами | `ConstraintError` сразу ([05](05-dry-types-boundaries.md)) |
| Источник значений | в модели | один `Types::*` для модели/GraphQL/валидаций |
| Гибкость значений | число (порядок ломать опасно) | символы (не зависят от порядка) |
| Хелперы | `paid?`, `paid!` | `released?` (через enumerize) |

## 🆚 Почему здесь dry-types
**Rails enum:** удобен, но привязывает к числовому маппингу (нельзя свободно менять порядок), значения живут в модели, мусор на границе ловится не всегда.
**dry-types:** значения — символы, один `Types::*` переиспользуется в модели, GraphQL-скалярах и валидаторах, мусор бракуется на входе. Это и есть «тип как граница».

## Бенефиты dry-types-подхода
- Один источник значений на весь проект.
- Строгая валидация внешнего ввода.
- Значения не завязаны на числовой порядок в БД.

## Когда хватит Rails enum
Маленький проект, статус только внутри одной модели, не нужен общий источник значений → встроенный `enum` проще и достаточно.
