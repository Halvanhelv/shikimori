# 69 — Singleton-сервисы (один экземпляр на приложение)

## Простыми словами
Некоторые вещи нужны в одном экземпляре на всё приложение и держат данные в памяти: список бан-слов, гео-база. Их делают синглтоном — один объект, к которому все обращаются.

## Реальный код
```ruby
class Moderations::Banhammer
  include Singleton
  # держит список запрещённых слов/паттернов в памяти

  def abusive?(text)
    # проверка текста по загруженным правилам
  end
end

# использование откуда угодно:
Moderations::Banhammer.instance.abusive?(nickname)   # см. валидатор 42
```

## Как работает
- `include Singleton` (из stdlib) → у класса один метод `.instance`, всегда возвращает тот же объект.
- Данные (бан-слова, гео-база) грузятся раз при первом обращении и живут в памяти процесса.
- Все вызовы идут к одному объекту — не пересоздаём тяжёлое состояние.

## 🆚 Классический Rails
**Обычно:** либо создают объект на каждый вызов (`Banhammer.new.abusive?` — перезагружает правила каждый раз), либо суют в глобальную переменную/константу руками.
**Здесь:** `Singleton` гарантирует один экземпляр, ленивую инициализацию и общий доступ — без ручных глобалов.

## Бенефиты
- Тяжёлое состояние (словари, гео-базы) грузится один раз.
- Единая точка доступа, потокобезопасная инициализация.
- Родственник Repository ([09](09-repository-singleton.md)), но без обязательного «справочника по id».

## Когда не надо
Если состояния нет (чистая функция) — синглтон не нужен, хватит модульного метода. Синглтоны с мутабельным состоянием усложняют тесты — держи их read-only.
