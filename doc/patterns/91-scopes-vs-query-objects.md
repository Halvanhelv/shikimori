# 91 — Скоупы vs Query-объекты: что когда

## Простыми словами
Простой фильтр («только опубликованные») — это скоуп в модели. Сложный поиск с десятком условий, парсингом параметров и разными источниками — это отдельный query-объект. Граница: когда фильтрация начинает «жить своей жизнью», выноси её.

## Реальный код
```ruby
# СКОУП — для простого, в модели:
class Anime < ApplicationRecord
  scope :ongoing, -> { where(status: :ongoing) }
end
Anime.ongoing.limit(10)

# QUERY-ОБЪЕКТ — для сложного, отдельным классом (см. 07, 15):
Animes::Query.fetch(
  scope: Anime.all,
  params: { season: 'summer_2024', genre: '1,5', status: 'ongoing', ... },
  user: current_user
)
```

## Как работает
- **Скоуп** — короткая именованная выборка, живёт в модели, чейнится (`.ongoing.recent`).
- **Query-объект** — отдельный класс ([07](07-query-object-base.md)), принимает много параметров, парсит их ([15](15-animes-query-dsl.md)), собирает сложный запрос, тестируется изолированно.

## 🆚 Когда что
| Признак | Скоуп | Query-объект |
|--|--|--|
| Сложность | 1-2 условия | много опциональных фильтров |
| Парсинг входа | нет | да (сезоны, диапазоны, отрицания) |
| Где живёт | в модели | отдельный класс |
| Переиспользование | внутри модели | между REST/GraphQL/автокомплитом |
| Тест | через модель | в изоляции |

**Антипаттерн:** скоуп на 40 строк с кучей `if` внутри модели — это сигнал «пора в query-объект».

## Бенефиты query-объекта
- Модель не пухнет от сложных выборок.
- Сложный фильтр тестируется без модели.
- Переиспользуется между разными API.

## Когда не надо
Не выноси `where(active: true)` в отдельный класс — это перебор. Скоуп проще и читаемее для простого. Query-объект окупается на **сложной** фильтрации.
