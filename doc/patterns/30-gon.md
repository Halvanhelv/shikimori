# 30 — `gon` (Rails → JS)

## Простыми словами
Помогает «передать» данные из бэкенда на Ruby в код фронтенда на JavaScript — без ручной возни. Ты на сервере складываешь значения в `gon`, а в браузере читаешь их как обычный JS-объект `gon.foo`. Это как записка, которую сервер кладёт в карман странице, а скрипт потом достаёт и читает.

## Что
Передача переменных из Rails в JavaScript без ручной сериализации в data-атрибуты или инлайн-скрипты.

## Реальный код
```ruby
# в контроллере
def js_export
  gon.push is_favoured: @resource.favoured?
end

# контесты, критики, диалоги — прокидывают данные для фронта
gon.push votes: contest.votes_for(current_user)
```

```javascript
// в JS доступно как:
if (gon.is_favoured) { ... }
```

## Как работает
- `gon.push(hash)` собирает данные на стороне Rails.
- Гем рендерит их в `<script>window.gon = {...}</script>` в layout.
- JS читает `gon.foo` — без ручного JSON и data-атрибутов.

## Зачем / где полезно
Серверные данные нужны JS при загрузке страницы (флаги, id текущего юзера, конфиг). Чище, чем `data-*` россыпью или инлайн-JSON. Безопаснее — гем экранирует.

## 🆚 Классический Rails
**Обычно:** серверные данные прокидывают в JS вручную — через россыпь `data-*` атрибутов в разметке или инлайн-`<script>` с самописным JSON, который легко забыть заэкранировать.
**Здесь:** `gon.push(...)` на сервере, а гем сам рендерит `window.gon = {...}` и экранирует значения; в JS просто читаешь `gon.foo`.

## Бенефиты
- Нет ручной сериализации в JSON и россыпи `data-*`.
- Гем экранирует данные — меньше риск XSS.
- Единая «точка» данных для фронта вместо разбросанных атрибутов.
- Удобно прокидывать флаги, id текущего юзера, конфиг при загрузке страницы.

⚠️ Не клади чувствительное — `gon` виден в исходнике страницы.
