# 20 — `sidekiq-limit_fetch` (лимит конкуренции очередей)

## Простыми словами
Можно ограничить, сколько задач из одной очереди работает одновременно, чтобы тяжёлые из них не съели все ресурсы. Это как касса с табличкой «не более 2 человек одновременно»: остальные ждут своей очереди.

## Что
Ограничивает, сколько задач из конкретной очереди выполняется одновременно. Защищает тяжёлые ресурсы.

## Реальный код
```yaml
# config/sidekiq.yml
:limits:
  cpu_intensive: 2        # максимум 2 параллельно
  torrents_parsers: 1     # строго по одному
  webm_thumbnails: 1
  mal_parsers: 40
  achievements: 50
  imports: 2
  dangerous_actions: 1
```

## Как работает
- Глобальный лимит на очередь поверх обычной конкуренции Sidekiq.
- `torrents_parsers: 1` → парсер торрентов всегда один в моменте (внешний источник не любит параллель).
- `cpu_intensive: 2` → тяжёлые расчёты не съедят все треды.

## Зачем / где полезно
- Внешние API с лимитом коннектов.
- CPU/IO-тяжёлые задачи, которые нельзя пускать пачкой.
- Приоритизация: лёгкие очереди с большим лимитом, тяжёлые с малым.

Принцип: разделяй работу по очередям и дай каждой свой потолок пропускной способности.

## 🆚 Классический Rails
**Обычно:** в Sidekiq задают общее число тредов на воркер — и любая очередь может занять их все, забив параллельными парсерами или тяжёлыми расчётами.
**Здесь:** через `:limits` в `sidekiq.yml` каждой очереди задан свой потолок одновременных задач (парсер торрентов — строго 1, тяжёлые расчёты — 2).

## Бенефиты
- Тяжёлые задачи не вытесняют лёгкие из тредов.
- Уважение к лимитам внешних API (один коннект за раз).
- Приоритизация: лёгким очередям — большой лимит, тяжёлым — малый.
- Настройка декларативная, в одном yml-файле.
