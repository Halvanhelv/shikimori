# 87 — Faye-расширения (middleware для сообщений)

## Простыми словами
На websocket-сервере каждое входящее/исходящее сообщение проходит через «расширения» — перехватчики. Один проверяет токен (можно ли публиковать), другой логирует. Это middleware, но для сообщений, а не HTTP-запросов.

## Реальный код
```javascript
// faye-server/server.js
var serverAuth = {
  incoming: function (message, callback) {
    if (message.channel.match(/^\/meta\//)) return callback(message);  // служебное — пропускаем
    if (message.data && message.data.token == key) {
      delete message.data.token;          // токен ок → вырезаем и пропускаем
    } else {
      message.error = 'No client posting is allowed';   // нет токена → блок
    }
    callback(message);                    // передать дальше по цепочке
  }
};
bayeux.addExtension(serverAuth);
bayeux.addExtension(fayeLogger);          // ещё одно расширение — логирование
```

## Как работает
- Расширение — объект с `incoming`/`outgoing` функциями.
- Каждое сообщение проходит через все расширения по очереди; каждое вызывает `callback(message)`, чтобы передать дальше.
- `serverAuth` гейтит публикации (только сервер с токеном), `fayeLogger` пишет лог.
- Расширения складываются (`addExtension`) как звенья.

## 🆚 Классический подход
**Обычно (наивно):** проверку токена и лог вписали бы внутрь каждого обработчика сообщения → дублирование, легко забыть.
**Здесь:** сквозные заботы (auth, лог) вынесены в расширения, применяются ко **всем** сообщениям автоматически. Та же идея, что Rack-middleware ([86](86-plug-pipeline.md)), но для websocket-сообщений.

## Бенефиты
- Auth/лог в одном месте, для всех сообщений.
- Расширения складываются и переиспользуются.
- Обработчики каналов чистые — без сквозной логики.

## Когда полезно вне Faye
Любая шина сообщений/событий: выноси аутентификацию, логирование, валидацию в перехватчики, а не в каждый хендлер.
