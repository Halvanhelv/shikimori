# 97 — Конфиг-маппинг как константа

## Простыми словами
Соответствия «тип → значение» (какой форум для аниме, какой для конкурсов) хранят как замороженную константу-хеш в коде, а не как магические числа, разбросанные по проекту.

## Реальный код
```ruby
class Topic < ApplicationRecord
  FORUM_IDS = {
    'Anime'   => 1,
    'Manga'   => 1,
    'Contest' => Forum::CONTESTS_ID,
    'Cosplay' => 6,
    # ...
  }.freeze

  def forum_id = FORUM_IDS[linked_type] || DEFAULT_FORUM_ID
end
```

## Как работает
- Маппинг объявлен один раз как `FROZEN` хеш-константа.
- Везде, где нужно «форум для типа X», читают `FORUM_IDS[type]`, а не пишут число.
- `.freeze` защищает от случайной мутации в рантайме.

## 🆚 Классический подход
**Обычно (плохо):** магические числа по коду — `forum_id = 1` тут, `if forum_id == 6` там. Поменялся форум — ищи все вхождения, что-то пропустишь.
**Здесь:** одна таблица соответствий. Поменять — в одном месте. Читается как документация («какие типы куда»).

## Бенефиты
- Один источник правды вместо магических чисел.
- Самодокументируемо (видно все соответствия сразу).
- `.freeze` — нельзя случайно испортить.

## Когда не надо
Если маппинг большой/часто меняется не-разработчиками — лучше БД/конфиг-файл ([09](09-repository-singleton.md) грузит из YAML). Константа хороша для небольших стабильных соответствий, известных на этапе кода.
