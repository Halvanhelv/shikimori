# 72 — Повторы при сбоях (retry в воркерах)

## Простыми словами
Внешний сервис временно недоступен — не теряем задачу, а ставим её на повтор через минуту. Сетевые сбои — норма, код должен их переживать.

## Реальный код
```ruby
class Achievements::Track
  def perform(user_id, user_rate_id, action)
    RedisMutex.with_lock("achievements/track_#{user_id}", block: 0, expire: 60) do
      Neko::Update.call(...)
    end
  rescue *(Network::FaradayGet::NET_ERRORS + [Neko::RequestError])
    self.class.perform_in 1.minute, user_id, user_rate_id, action   # сеть упала → повтор позже
  rescue RedisMutex::LockError
    self.class.perform_in 5.seconds, user_id, user_rate_id, action  # занято → повтор скоро
  end
end
```

## Как работает
- Ловим **конкретные** ошибки (сетевые, лок занят), а не все подряд.
- На сбой — переставляем ту же задачу через `perform_in(интервал)`.
- Разные причины → разные интервалы (сеть: минута; занятый лок: 5 секунд).

## 🆚 Классический Rails
**Обычно:** полагаются на дефолтный retry Sidekiq (экспоненциальный, для всех ошибок одинаково) или не обрабатывают вовсе → задача падает в dead-очередь.
**Здесь:** осознанный повтор под конкретную причину со своим интервалом; для расчёта ачивок `retry: false` + ручной `perform_in` — полный контроль.

## Бенефиты
- Не теряем работу при временных сбоях внешних сервисов.
- Разная стратегия под разную причину сбоя.
- Не забиваем очередь бессмысленными повторами на «настоящих» ошибках (ловим только ожидаемые).

## Когда не надо
Не делай «голый `rescue => e; retry`» — это маскирует баги и крутит бесконечно. Повторяй только **ожидаемые временные** ошибки, с лимитом/интервалом. Для простых случаев — встроенный retry Sidekiq/ActiveJob.
