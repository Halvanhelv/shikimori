# 85 — Сменный адаптер через @behaviour (Elixir)

## Простыми словами
neko общается с shikimori через «контракт»: список методов, которые должен реализовать клиент. Настоящий клиент ходит по HTTP, а в тестах подставляют фейковый — код не меняется, потому что оба соблюдают контракт.

## Реальный код
```elixir
defmodule Neko.Shikimori.Client do
  defmodule Behaviour do
    @callback get_user_rates!(pos_integer) :: [Neko.UserRate.t()]   # контракт
    @callback get_achievements!(pos_integer) :: [Neko.Achievement.t()]
    @callback get_animes!() :: [Neko.Anime.t()]
  end
  @behaviour Behaviour

  @adapter Application.get_env(:neko, :shikimori)[:client]   # какой клиент — из конфига
  defdelegate get_user_rates!(user_id), to: @adapter         # делегируем реальному адаптеру
end
```

## Как работает
- `@callback` объявляет контракт (какие функции и с какими типами обязательны).
- `@adapter` берётся из конфига → в проде это `HTTPClient`, в тестах — мок.
- `defdelegate` пробрасывает вызовы выбранному адаптеру.
- Компилятор проверяет, что адаптер реализует все `@callback`.

## 🆚 Классический подход
**Обычно (жёстко):** код напрямую дёргает `HTTPoison.get("shikimori.one/...")` → в тестах нельзя без реальной сети, нельзя подменить источник.
**Здесь:** зависимость на **контракт**, реализация выбирается конфигом. Тест подставляет фейк, прод — HTTP. Это Dependency Injection. Ruby-аналог — duck typing + передача зависимости в конструктор.

## Бенефиты
- Тесты без сети (подменяем адаптер).
- Можно сменить источник данных, не трогая вызовы.
- Компилятор гарантирует, что адаптер реализует контракт.

## Когда полезно вне neko
Любая внешняя зависимость (платёж, почта, API): объяви интерфейс, внедряй реализацию. В Rails — передавай клиента параметром (как `ai_client:` в gnosis), а не хардкодь.
