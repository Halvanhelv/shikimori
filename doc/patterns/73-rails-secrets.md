# 73 — Секреты и конфиг вне кода

## Простыми словами
Адреса сервисов, токены, ключи не пишут прямо в коде — их берут из секретов/ENV. Один и тот же код работает в dev и проде с разными значениями, а ключи не утекают в git.

## Реальный код
```ruby
# faye_publisher.rb — хост и токен из секретов, не захардкожены
def self.faye_url
  faye_host = Rails.application.secrets[:faye][:host].gsub('%DOMAIN%', shiki_domain)
  Shikimori::PROTOCOL + '://' + faye_host + Rails.application.secrets[:faye][:endpoint_path]
end

faye_client.publish(channel, data.merge(token: Rails.application.secrets.faye[:token]))

# Procfile (dev) задаёт переменные окружения:
# faye: FAYE_PORT=9292 FAYE_KEY=xxxxx node server.js
```

## Как работает
- Значения лежат в `secrets`/`credentials`/ENV, а не в коде.
- Подстановка `%DOMAIN%` → шаблон хоста параметризуется под окружение.
- dev и прод читают одни и те же ключи конфига, но с разными значениями.

## 🆚 Классический Rails
**Обычно (плохо):** `FAYE_HOST = "localhost:9292"` прямо в коде → в проде неверно, а реальный токен случайно уезжает в git.
**Здесь:** код ссылается на ключ конфига (`secrets.faye.host`), значение приходит из окружения. Rails 5.2+ даёт зашифрованные `credentials` (ключ — вне репозитория).

## Бенефиты
- Один код — много окружений (dev/test/prod).
- Секреты не в репозитории (зашифрованы/в ENV).
- Поменять хост/токен — без правки кода и редеплоя кода.

## Когда не надо
Неизменные не-секретные константы (например математические) можно держать в коде. В git — никогда реальные ключи; в dev-конфиге допустимы заглушки (`token: 'foo'`, см. [55](55-service-tokens.md)), но не настоящие секреты.
