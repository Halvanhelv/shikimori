# 60 — Декораторы (Draper) вместо хелперов

## Простыми словами
Декоратор — это «обёртка» вокруг модели, куда складывают всю логику отображения (как показать дату, аватар, ссылку). Модель остаётся про данные, вьюха — про вёрстку, а «как красиво показать» живёт в декораторе.

## Реальный код
```ruby
class BaseDecorator < Draper::Decorator
  include Translation
  delegate_all                       # все методы модели доступны через декоратор

  def self.inherited(target)
    target.prepend ActiveCacher.instance   # кеш дорогих методов (см. 03)
    super
  end
end

# использование:
anime = Anime.find(1).decorate
anime.name              # метод модели (делегирован)
anime.headline          # метод декоратора (форматирование для показа)
```

## Как работает
- `Draper::Decorator` оборачивает объект (`object`), `delegate_all` пробрасывает к нему все методы модели.
- Добавляешь методы представления (`headline`, `cover_url`) — они доступны рядом с модельными.
- В контроллере/вьюхе зовёшь `.decorate`, дальше работаешь с декоратором.

## 🆚 Классический Rails
**Обычно:** логику показа суют в **хелперы** (`app/helpers`) — это глобальные модули, методы висят в воздухе, не привязаны к объекту, легко конфликтуют именами, тяжело тестировать.
**Здесь:** логика показа привязана к конкретному типу (`AnimeDecorator`), вызывается как метод объекта (`anime.headline`), тестируется изолированно.

## Бенефиты
- Нет «свалки» глобальных хелперов.
- Презентация рядом с объектом, а не размазана.
- Не засоряет модель методами для вьюх (модель остаётся про данные/правила).
- Кеш дорогих вычислений из коробки (`ActiveCacher`).

## Когда не надо
Простой проект/одна-две вьюхи — хватит хелперов. В Rails 8 ту же роль играет ViewComponent (см. [43](43-view-objects.md)).
