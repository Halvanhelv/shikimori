# 90 — Подводный камень мемоизации (relation → array)

## Простыми словами
Когда кешируешь результат метода, важно кешировать **данные**, а не «ленивый запрос». ActiveRecord-связь — это не данные, а «обещание сходить в БД». Если закешировать обещание, оно может сходить в БД снова. Поэтому при кешировании связь превращают в массив.

## Реальный код
```ruby
# lib/active_cacher.rb — кешер сам конвертит relation в массив:
def prepare_for_cache(object)
  if object.is_a? ActiveRecord::Relation
    object.to_a        # материализуем: выполняем запрос, кешируем РЕЗУЛЬТАТ
  else
    object
  end
end
```

## Как работает
- `instance_cache :roles` ([03](03-instance-cache-prepend.md)) кеширует то, что вернул метод.
- Если метод вернул `Relation` (ленивый запрос), кешер вызывает `.to_a` → выполняет запрос **один раз** и кеширует массив записей.
- Иначе при каждом обращении к закешированной «связи» могли бы заново улетать запросы в БД.

## 🆚 Наивная мемоизация
**Обычно (баг):**
```ruby
def roles
  @roles ||= object.person_roles.where(...)   # кешируем RELATION, не данные!
end
```
`@roles` — ленивый relation. Каждое `roles.each`, `roles.size`, `roles.map` может бить в БД заново — мемоизация «не работает».
**Здесь:** кешер материализует (`to_a`) → запрос один раз, дальше из памяти.

## Бенефиты
- Кеш реально кеширует (один запрос вместо многих).
- Не нужно помнить про `.to_a` в каждом методе — кешер делает сам.

## Когда важно знать
Этот подводный камень есть в любой Rails-мемоизации связей. Правило: кешируешь связь — материализуй (`.to_a`/`.load`). Кешируешь скаляр — можно как есть.
