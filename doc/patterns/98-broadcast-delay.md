# 98 — Отложенный бродкаст (дать данным «дозреть»)

## Простыми словами
Когда создаётся аниме, форум-тему о нём не публикуют мгновенно — постер ещё качается в фоне, данные неполные. Бродкаст откладывают на несколько минут, чтобы опубликовать уже «дозревшую» запись.

## Реальный код
```ruby
# тема генерится, но рассылка подписчикам откладывается:
BroadcastTopicJob.set(wait: 10.minutes).perform_later(topic.id)
# или sidekiq:
SomeBroadcast.perform_in(10.minutes, topic.id)
```

## Как работает
- Сразу создаём запись (тему), но **не** шлём уведомления.
- Бродкаст ставится в фон с задержкой (`wait`/`perform_in`).
- За это время фоновые задачи дочиняют запись (постер, метаданные).
- Через N минут уведомление уходит уже по полной записи.

## 🆚 Классический подход
**Обычно (наивно):** шлют уведомление прямо в колбэке `after_create` → подписчики получают «полупустую» запись (без постера, без описания), плюс уведомление внутри транзакции создания может откатиться/задвоиться.
**Здесь:** уведомление отвязано от момента создания, ждёт готовности данных, идёт из фона (вне транзакции).

## Бенефиты
- Уведомляем о «готовой» записи, а не полуфабрикате.
- Бродкаст вне транзакции создания (нет задвоений при откате).
- Сглаживание нагрузки (рассылка не в момент пикового создания).

## Когда не надо
Realtime-события (новый коммент в открытом чате) шлют **сразу** — задержка вредна ([24](24-faye-realtime.md)). Отложенный бродкаст — для случаев, где данные дозревают в фоне. Подбирай задержку под реальное время готовности.
