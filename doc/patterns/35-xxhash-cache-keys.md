# 35 — `xxhash` для cache-key

## Простыми словами
Чтобы понять, изменился ли большой кусок текста (и пора ли сбросить кеш), считают его короткий «отпечаток» — хеш. `xxhash` делает это в разы быстрее, чем MD5/SHA, потому что тут не нужна защита от взлома — нужна только скорость и уникальность отпечатка. Это как штрихкод на товаре: быстро отличить один от другого, а не зашифровать секрет.

## Что
Очень быстрый некриптографический хеш для построения ключей кеша по большому тексту.

## Реальный код
```ruby
# app/decorators/db_entry_decorator.rb
cache_key = CacheHelperInstance.cache_keys(
  object.cache_key,
  XXhash.xxh32(text),   # быстрый хеш описания (может быть длинным)
  CACHE_VERSION
)
```

## Как работает
- `XXhash.xxh32(text)` — хеширует текст в число на порядки быстрее, чем MD5/SHA.
- Входит в составной ключ кеша: меняется текст → меняется хеш → инвалидация фрагмента.
- Криптостойкость тут не нужна (не защита, а идентификация контента), поэтому скорость важнее.

## Зачем / где полезно
Когда часть ключа кеша — большой/изменчивый контент (HTML, описание, JSON). MD5 на каждый рендер дорог. xxhash даёт ту же роль «отпечатка» за копейки CPU.

## 🆚 Классический Rails
**Обычно:** для отпечатка контента в ключе кеша по привычке берут `Digest::MD5`/`SHA` — это криптографические хеши, они надёжны, но заметно медленнее, а криптостойкость для кеша вообще не нужна.
**Здесь:** для той же задачи используется некриптографический `XXhash.xxh32`, который считает отпечаток на порядки быстрее.

## Бенефиты
- Хеширование большого текста почти бесплатно по CPU.
- Та же роль «отпечатка контента» для инвалидации кеша, что и у MD5.
- Меньше нагрузка при частом рендере фрагментов с большим контентом.

⚠️ Только там, где не нужна криптостойкость (не для подписей/паролей — для них см. HMAC в [23](23-camo-image-proxy.md)).
