# 36 — `ar_lazy_preload` (авто-фикс N+1)

## Простыми словами
Лечит классическую проблему N+1 (когда на список из 100 записей делается 100 лишних запросов за связанными данными) автоматически, без ручного перечисления `.includes`. Как только у одной записи дёргают связь, гем тут же подгружает её сразу для всех записей коллекции одним запросом. Это как заказать кофе на всю компанию за раз, а не бегать к стойке отдельно за каждым.

## Что
Автоматическая ленивая подгрузка ассоциаций — лечит N+1 без ручных `.includes`, особенно удобно в GraphQL.

## Реальный код
```ruby
# в GraphQL-резолверах
scope.lazy_preload(*PRELOADS)

# в QueryObjectBase lazy_preload добавляется только если гем есть:
QUERY_METHODS = %i[joins includes where ...] +
  (defined?(ArLazyPreload) ? %i[lazy_preload] : [])
```

## Как работает
- Помечаешь записи как «лениво-преload'имые».
- Когда у первой записи дёргают ассоциацию, гем подгружает её сразу для **всех** записей коллекции одним запросом.
- В GraphQL это спасает: заранее не знаешь, какие поля запросит клиент → нельзя статично прописать `includes`.

## Зачем / где полезно
- GraphQL/динамические API, где набор нужных ассоциаций неизвестен заранее.
- Глубокие графы связей, где ручной `includes` хрупок.
- Учит: N+1 можно лечить декларативно/лениво, а не только статичным eager-load.

## 🆚 Классический Rails
**Обычно:** N+1 лечат вручную через `.includes(:foo, :bar)` — но надо заранее знать, какие именно ассоциации понадобятся; забыл одну — снова N+1, добавил лишнюю — грузишь зря.
**Здесь:** `ar_lazy_preload` подгружает ассоциации лениво и автоматически для всей коллекции при первом обращении — особенно ценно в GraphQL, где набор полей заранее неизвестен.

## Бенефиты
- N+1 лечится сам, без ручного перечисления `.includes`.
- Работает даже когда набор нужных ассоциаций неизвестен заранее (GraphQL, динамические API).
- Меньше хрупкости на глубоких графах связей, чем у ручного eager-load.
- Не грузит лишнего: подгрузка происходит только при реальном обращении к связи.
