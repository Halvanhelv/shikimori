# 96 — has_many :through с «богатой» join-моделью

## Простыми словами
Когда связь многие-ко-многим имеет **свои данные** (не просто «связаны», а «связаны вот так»), join-таблица становится полноценной моделью. Пример: роль человека в аниме — это связь person↔anime, но с атрибутом «какая роль» (режиссёр, сэйю).

## Реальный код
```ruby
class PersonRole < ApplicationRecord     # «богатая» join-модель
  belongs_to :person
  belongs_to :anime
  belongs_to :character, optional: true
  # + своё поле: roles/role-type
end

class Anime < ApplicationRecord
  has_many :person_roles
  has_many :people, through: :person_roles   # многие-ко-многим ЧЕРЕЗ join-модель
end

anime.person_roles.first.character   # доступ к данным самой связи
```

## Как работает
- Join-модель (`PersonRole`) — обычная модель с собственными полями и связями.
- `has_many :through` даёт удобный доступ к «дальней» стороне (`anime.people`).
- Но через саму join-модель доступны атрибуты связи (роль, персонаж).

## 🆚 Простой `has_and_belongs_to_many` (HABTM)
**HABTM:** только связь «A связан с B», без своих полей. Join-таблица «немая».
**Здесь (`through` + модель):** у связи есть данные (тип роли, персонаж, время), своя валидация, свои колбэки. Нельзя выразить через немую HABTM.

## Бенефиты
- У связи есть собственные атрибуты и логика.
- Join-модель — обычная модель: валидации, скоупы, колбэки.
- Гибче, чем HABTM, который без полей.

## Когда не надо
Если связь действительно «немая» (просто A↔B, без данных) — `has_and_belongs_to_many` или массив id ([67](67-postgres-array-columns.md)) проще. `through` нужен, когда у связи есть **свои** атрибуты.
