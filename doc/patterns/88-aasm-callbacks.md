# 88 — AASM-колбэки переходов (after / success / guard)

## Простыми словами
У конечного автомата ([22](22-aasm-state-machines.md)) на каждый переход можно навесить действия: проверить, можно ли переходить (guard), что сделать при переходе (after), что после успеха (success). Логика «что происходит при смене статуса» собрана в описании автомата.

## Реальный код
```ruby
class Version < ApplicationRecord
  aasm column: 'state' do
    event :accept do
      transitions from: :pending, to: :accepted do
        guard   { can_be_accepted? }     # МОЖНО ли переходить
        after   :apply_version            # применить дифф к записи (см. 12)
        after   :assign_moderator
        success :notify_acceptance        # уведомить после успеха
      end
    end
  end
end

version.accept!   # проверит guard → сменит state → after → success
```

## Как работает
- `guard` — переход выполнится, только если вернул true (иначе ошибка/отказ).
- `after` — действия при переходе (может быть несколько, по порядку).
- `success` — после успешного сохранения нового состояния.
- Всё это срабатывает на `version.accept!`.

## 🆚 Классический Rails
**Обычно:** на смену статуса вешают `after_save`/`before_save` колбэки модели с `if status_changed? && status == 'accepted'` — лапша условий, срабатывают не только на нужный переход, трудно понять «что будет при accept».
**Здесь:** действия привязаны к **конкретному переходу** (`pending → accepted`), а не к «любому сохранению». Видно весь жизненный цикл в одном блоке.

## Бенефиты
- Действия привязаны к переходу, а не к «save» вообще.
- Запрет невалидных переходов через `guard`.
- Весь lifecycle читается в описании автомата.

## Когда не надо
Много тяжёлой логики в `after` → лучше вынести в операцию ([81](81-operations-composition.md)) и звать её из колбэка. Колбэки не должны превращаться в простыню.
