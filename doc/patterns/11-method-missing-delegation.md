# 11 — `method_missing`-делегирование

## Простыми словами
В базе хранят только сухой факт («у юзера есть ачивка X уровня Y»), а всё остальное — название, картинку, описание — берут из общего справочника в памяти. Если у объекта спрашивают то, чего у него нет, он молча перенаправляет вопрос справочнику. Как сотрудник, который на любой незнакомый вопрос отвечает «уточню в инструкции» и зачитывает оттуда.

## Что
Тонкая модель хранит сухой факт в БД, а всё «как показать/что значит» делегирует через `method_missing` в объект-правило из памяти.

## Реальный код
```ruby
class Achievement < ApplicationRecord
  belongs_to :user
  delegate :cache_key, to: :neko

  def respond_to_missing?(*args) = super || neko.send(:respond_to_missing?, *args)
  def method_missing(method, *args, &block) = neko.send(method, *args, &block)

  private
  def neko = @neko ||= NekoRepository.instance.find(neko_id, level)
end
```

## Как работает
- В БД лежит только `neko_id` + `level` (факт «у юзера ачивка X уровня Y»).
- `title`, `image`, `hint`, `threshold` приходят из `Neko::Rule` (YAML-правило в памяти, см. [09](09-repository-singleton.md)).
- `method_missing` ловит любой неопределённый метод и шлёт его правилу.
- `respond_to_missing?` держит `respond_to?` честным (важно для Rails-хелперов, форм).

## Зачем / где полезно
Когда у записи мало собственных данных, но много производной логики из общего справочника/конфига. БД-схема остаётся тощей, поведение меняется правкой YAML, не миграцией.

⚠️ `method_missing` без `respond_to_missing?` ломает интроспекцию — всегда определяй оба.

## 🆚 Классический Rails
**Обычно:** все атрибуты ачивки (title, image, threshold) хранят колонками в таблице и дублируют для каждой записи. Изменение текста — это миграция или массовый апдейт.
**Здесь:** в БД только `neko_id` + `level`, а производные данные приходят из YAML-правила; `method_missing` прозрачно делегирует туда любой неизвестный вызов.

## Бенефиты
- Тощая БД-схема — только сам факт, без дублирования справочных данных.
- Поведение и тексты меняются правкой YAML, а не миграцией.
- Один справочник — источник правды для всех записей одного типа.
