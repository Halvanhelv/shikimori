# 61 — STI (наследование на одной таблице)

## Простыми словами
Несколько похожих типов хранятся в **одной** таблице, а различаются колонкой `type`. Например все темы форума лежат в `topics`, но `AnimeTopic`, `ClubTopic`, `NewsTopic` — это разные Ruby-классы с разным поведением.

## Реальный код
```ruby
class Topic < ApplicationRecord
  # колонка type определяет класс автоматически
end

class Topics::EntryTopics::AnimeTopic < Topic
  def updated_at = nil   # своё поведение: не «тонуть» от свежих комментов (см. 16)
end

# при выборке Rails сам отдаёт нужный подкласс:
Topic.find(1)            # => #<Topics::EntryTopics::AnimeTopic>
```

## Как работает
- В таблице есть колонка `type` (строка с именем класса).
- Создаёшь `AnimeTopic.create` → Rails пишет `type: "Topics::EntryTopics::AnimeTopic"`.
- Читаешь `Topic.all` → Rails возвращает объекты правильных подклассов, у каждого своё поведение.

## 🆚 Классический Rails
**Обычно (без STI):** заводят отдельные таблицы `anime_topics`, `club_topics`, `news_topics` — дублирование колонок, нельзя выбрать «все темы» одним запросом, общие связи (комментарии) приходится делать к каждой.
**Здесь (STI):** одна таблица + подклассы. Общий список одним запросом, общие связи к `Topic`, различия — в методах подклассов.

## Бенефиты
- Один запрос на «все темы любого типа».
- Общие ассоциации/скоупы наследуются.
- Различия поведения — через ООП (подклассы), а не `if type == ...`.

## Когда не надо
Если у типов сильно разные поля → таблица распухнет от `NULL`-колонок. Тогда лучше отдельные таблицы или полиморфизм ([62](62-polymorphic-associations.md)).
