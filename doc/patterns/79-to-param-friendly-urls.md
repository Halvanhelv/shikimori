# 79 — Человекочитаемые URL (`to_param`)

## Простыми словами
Вместо `/animes/21` показывать `/animes/21-naruto` — красиво и полезно для SEO. Достигается переопределением одного метода модели.

## Реальный код
```ruby
class Anime < ApplicationRecord
  def to_param
    "#{id}-#{name.parameterize}"    # 21-naruto
  end
end

# поиск всё равно по id (Rails берёт ведущее число):
Anime.find(params[:id])   # params[:id] = "21-naruto" → .to_i = 21
```

## Как работает
- `to_param` определяет, что подставится в URL вместо id.
- `link_to anime` и `anime_path(anime)` используют `to_param` → красивый slug.
- При чтении `params[:id]` = `"21-naruto"`, а `.to_i` = `21` → `find` находит по числу. Slug — декоративный.

## 🆚 Классический Rails
**Обычно:** `/animes/21` — id в чистом виде, не информативно, хуже для SEO/шаринга.
**Здесь:** `/animes/21-naruto` без отдельной колонки-слага и без лишних запросов (id в начале строки). Альтернатива — гем `friendly_id` (slug в колонке, поиск по слагу).

## Бенефиты
- Читаемые, SEO-дружелюбные ссылки.
- Не нужна отдельная колонка/индекс (slug выводится из имени).
- Ломать существующие `/21` ссылки не обязательно — id в начале.

## Когда не надо
Если нужен поиск **строго по слагу** (без id в URL, например `/users/morr`) — там лучше `friendly_id` с колонкой и индексом. `to_param` со слагом-после-id хорош, когда id остаётся ключом.
