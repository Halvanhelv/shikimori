# 65 — Search-объекты (запросы к Elasticsearch)

## Простыми словами
Каждый вид поиска (аниме, манга, персонажи) — отдельный объект-функция. Контроллер просто зовёт `Search::Anime.call(phrase: ...)`, а вся магия запроса к Elasticsearch спрятана внутри.

## Реальный код
```ruby
# app/query_objects/search/anime.rb
Search::Anime.call(phrase: 'narto', ids_limit: 10)

# контроллер автокомплита — тонкий (см. 50):
def index
  render json: Search::Fansubber.call(phrase: search_phrase, ids_limit: LIMIT, kind: params[:kind])
end
```

## Как работает
- Search-объект — callable ([01](01-method-object.md)): принимает фразу/лимиты, строит запрос к Chewy/Elasticsearch ([18](18-chewy-elasticsearch.md)), возвращает результаты.
- Логика поиска (анализаторы, скоринг, нормализация) инкапсулирована — снаружи только `phrase` и `limit`.
- Каждая сущность — свой Search-класс (`Search::Anime`, `Search::Manga`, `Search::Character`).

## 🆚 Классический Rails
**Обычно:** поисковый запрос пишут прямо в контроллере или в скоупе модели — он разрастается (фильтры, скоринг, fallback), дублируется между REST/GraphQL/автокомплитом.
**Здесь:** поиск — отдельный объект, переиспользуется везде, тестируется без HTTP, не раздувает модель/контроллер.

## Бенефиты
- Одна точка правды поиска по сущности.
- Тонкие контроллеры (input → Search-объект → JSON).
- Легко поменять движок (ES → другой) внутри объекта, не трогая вызовы.

## Когда не надо
Простой `WHERE name ILIKE` — оставь скоуп в модели. Search-объект окупается, когда поиск нетривиален (ES, скоринг, много фильтров).
