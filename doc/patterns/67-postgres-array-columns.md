# 67 — Массивы PostgreSQL в колонке

## Простыми словами
Вместо отдельной таблицы-связки можно хранить список id прямо в колонке-массиве. У аниме `genre_ids = [1, 5, 12]` лежит в одной строке, и можно искать «аниме с жанром 5» прямо по массиву.

## Реальный код
```ruby
# фильтр каталога (см. 15): пересечение массивов через оператор &&
scope.where("genre_ids && '{#{genre_ids.join(',')}}'")
# && = "массивы пересекаются" → есть хотя бы один общий жанр

# аналогично для жанров v2:
scope.where! "genre_v2_ids && '{#{ids}}'"
```

## Как работает
- Колонка типа `integer[]` хранит массив прямо в строке.
- Оператор `&&` PostgreSQL = «массивы имеют общий элемент».
- `genre_ids && '{5,12}'` → true, если у аниме есть жанр 5 ИЛИ 12.
- Можно навесить GIN-индекс → быстрый поиск по массиву.

## 🆚 Классический Rails
**Обычно:** делают join-таблицу `anime_genres (anime_id, genre_id)` + `has_many :through`. Для «аниме с жанром X» — JOIN, для нескольких жанров — JOIN + GROUP/DISTINCT, запросы тяжелеют.
**Здесь:** массив в колонке + `&&` → без JOIN, один индекс, простой `WHERE`.

## Бенефиты
- Нет join-таблицы и JOIN'ов для частого «есть ли жанр».
- GIN-индекс делает поиск по массиву быстрым.
- Меньше строк в БД, проще выборка.

## Когда не надо
Если у связи есть **свои атрибуты** (например «когда добавлен жанр», «кем») — нужна полноценная join-модель. Массив хорош для простого «список id без метаданных». Также теряешь FK-целостность (БД не проверит, что genre_id существует).
