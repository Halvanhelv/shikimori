# 81 — Операции, вызывающие операции (композиция)

## Простыми словами
Сложное действие собирается из маленьких. «Засчитать вышедшую серию» внутри вызывает «отследить уведомление о серии», та — ещё мельче. Каждый кусок — отдельный callable-объект, и они складываются как кубики.

## Реальный код
```ruby
class Anime::IncrementEpisode
  method_object %i[anime! aired_at! user!]

  def call
    # ... своя логика ...
    EpisodeNotification::Track.call(...)   # операция вызывает другую операцию
  end
end

class EpisodeNotification::Track
  method_object :anime
  def call
    EpisodeNotification::TrackEpisode.call(...)   # и она — следующую
  end
end
```

## Как работает
- Каждая операция — callable ([01](01-method-object.md)) с единственной ответственностью.
- Крупная операция оркеструет: зовёт мелкие по очереди.
- Получается дерево: верхняя задаёт сценарий, нижние делают конкретные шаги.

## 🆚 Классический Rails
**Обычно:** всё суют в один «толстый» метод модели/контроллера на 100 строк (`def increment_episode; ...; end`) — невозможно переиспользовать кусок и тяжело тестировать.
**Здесь:** каждый шаг — отдельный объект. Можно вызвать `EpisodeNotification::Track` сам по себе, протестировать изолированно, переиспользовать в другом сценарии.

## Бенефиты
- Маленькие тестируемые куски вместо монолита.
- Переиспользование шагов между сценариями.
- Сценарий читается как список вызовов, а не простыня кода.

## Когда не надо
Не дроби ради дробления: 3 строки логики не нужно разносить на 3 класса. Композиция окупается, когда шаги реально самостоятельны и переиспользуемы.
