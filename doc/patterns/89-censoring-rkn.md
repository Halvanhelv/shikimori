# 89 — Цензура/RKN-фильтрация (юридические ограничения)

## Простыми словами
По закону часть контента нельзя показывать в определённых странах. Это вынесено в concern и cross-cutting фильтр, чтобы «спрятать запрещённое» работало единообразно, а не проверялось руками в каждом месте.

## Реальный код
```ruby
class Anime < ApplicationRecord
  include RknConcern        # логика «забанено Роскомнадзором»
end

# контроллер подмешивает CensoredConcern (см. 45):
class ApplicationController < ActionController::Base
  include CensoredConcern   # вторым по порядку — определяет режим цензуры рано
end

# ошибки-сигналы для прерывания (см. 44):
class RknBanned < StandardError; end
class AgeRestricted < StandardError; end
```

## Как работает
- `RknConcern` помечает/определяет запрещённые записи.
- `CensoredConcern` в контроллере включает режим цензуры (на основе домена/страны) рано в цепочке.
- Где нельзя показать — выбрасывается типизированная ошибка (`RknBanned`), которую ловит `ErrorsConcern` и отдаёт нужный ответ.

## 🆚 Классический Rails
**Обычно:** проверку «можно ли показать» копируют в каждый контроллер/вьюху (`if anime.banned? && user.in_russia? ...`) → легко забыть в новом месте, юридический риск.
**Здесь:** сквозная политика через concern'ы + сигнальные ошибки. Единообразно, нельзя «забыть» — фильтр глобальный.

## Бенефиты
- Юридические ограничения в одном месте, не размазаны.
- Нельзя случайно пропустить запрещённый контент в новом экшене.
- Прерывание через ошибку (см. [44](44-domain-errors.md)) — чисто и централизованно обрабатывается.

## Когда полезно вне shikimori
Любые регуляторные/гео-ограничения (GDPR, возрастной контент, лицензии по регионам): делай сквозной фильтр + сигнальные исключения, а не проверки в каждом месте.
