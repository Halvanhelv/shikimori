# 43 — ViewObject (vs Decorator)

## Простыми словами
Декоратор оборачивает одну модель и красиво её показывает. Но бывают куски страницы (меню, реклама, дашборд), у которых нет «своей» модели — они собираются из конфига, политик и нескольких запросов сразу. Для них и нужен ViewObject — объект, который собирает целый блок из разных источников. Это как шеф-повар, который готовит блюдо из продуктов с разных полок, а не «украшает» один готовый ингредиент.

## Что
Объект представления, не привязанный к одной модели: композиция данных для куска страницы (меню, дашборд, реклама, лейаут). Легче декоратора.

## Реальный код
```ruby
class ViewObjectBase
  include Draper::ViewHelpers   # доступ к h. (хелперы вьюх)
  include Translation
  prepend ActiveCacher.instance # кеш дорогих методов (см. 03)

  def read_attribute_for_serialization(attribute) = send(attribute)
end

# пример: Ad — реклама (≈470 строк): правила показа + рендер баннеров
class Ad < ViewObjectBase
  def html
    return unless AdsPolicy.new(...).allowed?   # гейт через политику
    # сборка баннера по правилам
  end
end
```

## Как работает
- **Decorator** оборачивает **одну модель** (`object`) и добавляет ей презентацию (см. [39](39-decorator-caching.md)).
- **ViewObject** не привязан к модели — собирает представление из нескольких источников (конфиг, политики, запросы).
- `Draper::ViewHelpers` даёт доступ к `h.` (роуты, хелперы) вне вьюхи.
- Кеш дорогих вычислений через `prepend ActiveCacher`.

## Зачем / где полезно
Сложные блоки UI без «своей» модели: топ-меню, дашборд, лейаут, реклама, open graph. Логика представления уезжает из вьюх/хелперов в тестируемый объект. В Rails 8 эту роль закрывает ViewComponent.

## 🆚 Классический Rails
**Обычно:** логику сборки таких блоков пишут в хелперах (`app/helpers`) или прямо в шаблонах с кучей `if` — глобально, без привязки к объекту, тяжело тестировать.
**Здесь:** блок без своей модели собирается отдельным объектом-наследником `ViewObjectBase` (доступ к `h.`, переводам и кешу). Чёткое разделение: декоратор — для одной модели, ViewObject — для составного блока из многих источников.

## Бенефиты
- Сложный UI-блок инкапсулирован в один тестируемый объект.
- Вьюхи и хелперы худеют — логика уезжает из шаблонов.
- Доступ к роутам/хелперам (`h.`) и кешу дорогих методов из коробки.
- Чёткая граница «одна модель → декоратор», «составной блок → ViewObject».
